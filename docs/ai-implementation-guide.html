2. 请求完整流转路径

1用户在 AiPanel 输入消息，按回车发送
2useAiChat 发送 POST /llm/chat/stream，携带 { message, sessionId, model?, provider? }
3路由层通过 Zod 校验，设置 SSE 头（text/event-stream）
4初始化 AIService：new AIService(userId).init(provider)
5从数据库读取 AI 配置（API Key 解密、Base URL 解析），创建 OpenAI 客户端
6prompt-selector 根据关键词匹配选择系统提示词
7路由层追加工具清单提示 + 时间解释指令到系统提示词末尾
8从数据库加载最近 10 条历史消息作为上下文
9用户消息保存到数据库（Conversation 表）
10进入 Agentic 工具循环（最多 10 轮）：
• 发送流式请求给 LLM，携带全部 33 个工具定义
• 收集 text delta 和 tool_call delta
• 如有 tool_call → 通过 toolMap 查找并执行 → 结果截断到 500 字符 → 追加到消息
• 再次请求 LLM（带上工具结果）→ 循环直到无 tool_call
11流式事件通过 SSE 推送给前端：text / tool_call / tool_result / done
12AI 回复保存到数据库，发送 [DONE] 终止
13前端收到 done 后，如涉及写工具，自动刷新 React Query 缓存

3.1 提示词选择机制

prompt-selector.ts 使用正则关键词匹配，根据用户消息内容自动路由到最合适的系统提示词：

3.2 五个提示词详细说明

💬 system-default.txt — 通用助手（99 行，最全面）

定义角色为"TaskFlow+ 智能助手，帮助一人公司老板管理项目、任务、客户和财务"。

核心规则（7 条）：

  1. 如实回答 — 工具返回什么就说什么，不编造
  2. 工具策略 — 优先用工具查数据，不凭记忆回答
  3. 参数处理 — 有默认值直接填，必填字段无法推断才追问
  4. 写操作确认 — 先说明要做什么，等用户确认再执行
  5. 回复风格 — 简洁直接，用 Markdown，金额用 ¥ 格式
  6. 列表限制 — 不超过 5 条，超出用"共 X 条，前 5 条"
  7. 禁止行为 — 不编造数据、不过度追问、不用模糊回答
📝 system-create.txt — 创建管理助手（68 行）

专注于创建和管理操作，定义了 3 步工作流程：理解意图 → 智能填充 → 确认执行。

智能推断规则：

  • 项目推断：用户没指定项目 → 查找最近活跃项目自动关联
  • 优先级推断："紧急/马上/今天必须" → URGENT，"重要/优先" → HIGH，"不急" → LOW
  • 日期推断："今天" → 调用 get_current_time，"明天" → +1 天，"月底" → 当月最后一天

确认格式示例：

将创建以下任务，请确认：

📋 **任务标题**
- 项目：XX 项目
- 优先级：🔴 HIGH
- 工时：4h
- 截止：2026-06-05

确认创建？

4. 工具调用机制

4.1 ToolDefinition 接口

interface ToolDefinition {
  name: string;           // 工具唯一标识
  description: string;    // 中文描述，LLM 用来判断何时调用
  category: 'finance' | 'work' | 'client' | 'goal' | 'schedule';
  parameters: {
    type: 'object';
    properties: Record<string, {
      type: string;       // string / number / enum
      description?: string;
      enum?: string[];
      default?: unknown;
    }>;
    required?: string[];
  };
  handler: (args, userId) => Promise<unknown>;  // 实际执行函数
  access: 'read' | 'write';     // 读操作直接执行，写操作需确认
  requiresConfirmation: boolean; // 写工具均为 true
  preferredModel: 'fast' | 'balanced' | 'powerful';
}

4.2 Agentic 工具循环

AI 不是简单的问答，而是一个多轮工具调用循环：

1用户消息 + 系统提示词 + 历史上下文 → 发送给 LLM
2LLM 分析后决定：直接回答 或 调用工具
3如调用工具 → 后端执行 handler → 结果截断 500 字符 → 追加到消息
4LLM 看到工具结果后决定：继续调用其他工具 或 生成最终回答
5循环直到 LLM 不再调用工具（最多 10 轮）

4.3 工具注册表（registry.ts）

所有 33 个工具在 registry.ts 中统一注册，提供以下查询方法：

TOTAL_TOOLS                    // 33
getAllTools()                   // 返回全部工具
getTool(name)                  // 按名称查找
getToolsByCategory(category)   // 按分类筛选

8. 安全与限制

机制说明
API Key 加密AES-256-GCM，存储格式 iv:authTag:ciphertext
消息长度限制单条消息最长 2000 字符（Zod 校验）
上下文窗口自动截断，保留系统消息 + 最近 8 条（约 4000 token）
工具结果截断单个工具结果最长 500 字符，数组只保留前 5 条
工具循环上限最多 10 轮工具调用，防止无限循环
写操作确认所有写工具 requiresConfirmation: true，前端弹出确认对话框
错误脱敏sanitizeErrorMessage() 从错误信息中剥离 API Key
Mock 模式未配置 API Key 时自动降级为直接查询数据库
用户隔离所有数据查询均带 userId 过滤，不同用户数据完全隔离

9. 扩展指南

9.1 添加新工具

  1. 在 backend/src/ai/tools/ 下创建 new-tool.ts
  2. 实现 ToolDefinition 接口（name/description/parameters/handler/access）
  3. 在 registry.ts 的 allTools 数组中注册
  4. （可选）在相关 system-*.txt 提示词中提及新工具的用途

9.2 添加新提示词

  1. 在 backend/src/prompts/ 下创建 system-xxx.txt
  2. 在 prompt-selector.ts 的 routes 数组中添加关键词匹配规则

9.3 添加新供应商

  1. 在 setting.service.ts 的 PRESET_PROVIDERS 数组中添加预设
  2. （可选）在 getFallbackModels() 中添加该供应商的常见模型名
  3. 用户也可通过前端设置页面的"添加供应商"自行添加自定义供应商

9.4 用户记忆提取

memory-extractor.ts 使用正则从对话中提取用户偏好，存储到 UserMemory 表。当前支持：

  • 工作习惯：每天工作 N 小时
  • 优先级偏好：默认优先级
  • 回复风格：简洁/详细
  • 成本意识：关注报价/成本

置信度从 0.5 开始，每次重复检测 +0.1。可扩展正则模式以支持更多偏好类型。
