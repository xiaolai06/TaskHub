# 排期视图 — 功能说明与实现原理

> 本文档描述 TaskFlow+ 排期视图的完整功能、用户交互流程和底层实现机制。

---

## 一、功能概述

排期视图是一个 **甘特图式的项目排期工具**，核心能力：

| 能力 | 说明 |
|------|------|
| 自动排期 | 根据任务优先级、工时估算、截止日期，自动计算每天该做什么 |
| 甘特图展示 | 可视化时间轴，直观看到每个任务的起止时间和重叠情况 |
| 延期预警 | 自动检测排期超过截止日期的任务 |
| 冲突检测 | 识别某天工时超限（>8h）和任务时间重叠 |
| 插单模拟 | "如果临时加一个任务，会影响哪些任务？" — 事前评估 |
| AI 排期建议 | 通过 AI 助手询问排期相关问题（今日焦点、健康度、何时有空等） |
| 应用排期 | 一键将计算结果写回任务的实际日期字段 |
| 每日提醒 | 每天 8:00 自动检测过期任务并推送通知 |

## 二、用户视角 — 操作流程

### 2.1 进入排期视图

两种入口：
1. **侧边栏** → 点击「排期视图」→ 进入页面后选择项目
2. **项目详情页** → 点击排期按钮 → 跳转并自动选中该项目 (`/main/schedule?projectId=xxx`)

### 2.2 查看排期

进入页面后，系统自动完成以下计算：

```
选择项目 → 后端拉取该项目所有未完成任务
         → 按优先级+截止日期排序
         → 从今天开始逐日分配工时
         → 生成甘特图 + 统计面板
```

页面分为三个区域：

```
┌─────────────────────────────────────────────────┐
│  统计面板 (ScheduleStats)                         │
│  任务总数 | 排期跨度 | 日均工时 | 延期 | 冲突 | 预计完成  │
├────────────────────────┬────────────────────────┤
│  甘特图 (GanttChart)    │  操作面板               │
│                        │  - 工时限制调整          │
│  ┌──────┬───┬───┬───┐  │  - 插单模拟按钮          │
│  │任务A │██ │   │   │  │  - AI 排期按钮           │
│  │任务B │   │███│   │  │  - 应用排期按钮          │
│  │任务C │   │ ██│   │  │  - 工作量柱状图          │
│  └──────┴───┴───┴───┘  │                        │
│     今日 ↑              │                        │
│  每日工时: 6  8  5  ... │                        │
└────────────────────────┴────────────────────────┘
```

### 2.3 关键操作

**调整工时限制**：默认每天 8 小时，可调 1-24h。降低限制会让排期拉长，升高会让排期压缩。

**插单模拟**：
1. 点击「插单模拟」→ 填写新任务信息（标题、优先级、工时、截止日期）
2. 系统模拟加入新任务后重新排期
3. 显示影响报告：哪些任务被推迟了、项目整体延期几天

**应用排期**：
- 点击「应用排期到任务日期」→ 系统将甘特图上的计算结果写入每个任务的 `startDate` 和 `dueDate`
- 这是一个**实际写入数据库**的操作，不可撤销

**AI 排期**：
- 点击「AI 排期」→ 将当前排期数据发送给 AI 助手进行分析
- 也可以在 AI 侧边栏的「排期」标签页中使用预设问题（延期任务、今日焦点、排期健康度等）

## 三、实现原理 — 核心算法

### 3.1 调度算法 (`buildSchedule`)

这是整个排期系统的核心，位于 `backend/src/services/scheduler.service.ts`。

**输入**：
- 任务列表（每个任务有 `estimatedHours`、`priority`、`dueDate`、`startDate`）
- 每日工时上限（默认 8h）
- 是否跳过周末（默认跳过）

**算法流程**：

```
Step 1: 排序
  ┌─────────────────────────────────────────────┐
  │ 排序键 = (优先级权重, 截止日期, 工时估算)        │
  │                                               │
  │ 优先级权重: URGENT=0 > HIGH=1 > MEDIUM=2 > LOW=3 │
  │ 截止日期:   越早越靠前                          │
  │ 工时估算:   越小越靠前（快速完成优先）            │
  └─────────────────────────────────────────────┘

Step 2: 逐日分配
  ┌─────────────────────────────────────────────┐
  │ 游标 = 今天                                   │
  │                                               │
  │ for 每个任务:                                  │
  │   if 任务有 startDate 且 > 游标:               │
  │     游标 = startDate  （尊重任务的开始约束）      │
  │   if 游标是周末:                               │
  │     跳到下周一                                 │
  │                                               │
  │   剩余工时 = 任务.estimatedHours                │
  │   记录 scheduledStart = 游标                    │
  │                                               │
  │   while 剩余工时 > 0:                          │
  │     if 当天已用工时 + 剩余工时 ≤ 每日上限:       │
  │       全部分配到今天，结束                       │
  │     else:                                     │
  │       分配 (每日上限 - 当天已用工时)，游标+1天    │
  │       if 周末: 跳到下周一                       │
  │                                               │
  │   记录 scheduledEnd = 游标                      │
  │   游标 += 1天（下一个任务从次日开始）             │
  └─────────────────────────────────────────────┘

Step 3: 标记延期和冲突
  ┌─────────────────────────────────────────────┐
  │ if 任务有 dueDate:                            │
  │   if scheduledEnd > dueDate:                  │
  │     标记 isDelayed = true, delayDays = 差值     │
  │                                               │
  │ for 每个工作日:                                │
  │   if 当天总工时 > 每日上限:                     │
  │     标记 isOverloaded = true                   │
  └─────────────────────────────────────────────┘
```

**输出**：
- 每个任务的 `scheduledStart` / `scheduledEnd` / `isDelayed` / `delayDays` / `isConflict`
- 每天的 `dailyWorkload`（工时数 + 是否超载）
- 汇总统计（总任务数、总工时、项目起止日期）

### 3.2 冲突检测 (`detectConflicts`)

两种冲突类型：

```
类型 1: 时间重叠冲突
  两两比较有明确起止日期的任务:
    if 任务A 的日期范围 和 任务B 的日期范围 重叠:
      → 记录重叠冲突

类型 2: 工时超载冲突
  运行 calculateSchedule():
    if 某天总工时 > dailyHourLimit:
      → 记录超载日
```

### 3.3 插单模拟 (`insertionSimulation`)

```
┌──────────────────────────────────────┐
│ 1. 计算原始排期（原始任务列表）         │
│ 2. 创建虚拟任务 (id='__new_task__')    │
│ 3. 加入虚拟任务后重新排期              │
│ 4. 对比两次排期:                      │
│    - 哪些任务的结束日期被推迟了？       │
│    - 项目整体结束日期变化多少？         │
│    - 新任务被安排在什么位置？           │
│ 5. 生成影响报告                       │
└──────────────────────────────────────┘
```

## 四、技术架构

### 4.1 无独立数据库表

排期**不存储在数据库中**，每次打开页面都是实时计算。这样做的好处：
- 任务修改后排期自动更新，无需同步
- 不需要维护排期数据的一致性
- 算法是纯函数，输入确定则输出确定

"应用排期"操作例外 — 它会把计算结果写回 Task 表的 `startDate` / `dueDate`。

### 4.2 前后端分工

```
前端 (React + TypeScript)                    后端 (Express + Prisma)
┌─────────────────────────┐                 ┌──────────────────────────┐
│ useSchedule hook        │ ──POST /calculate──→ │ scheduler.service.ts   │
│   useQuery + 缓存       │                 │   buildSchedule() 纯算法  │
│                         │                 │                          │
│ GanttChart (自绘甘特图)  │ ← JSON 数据 ─── │ calculateSchedule()      │
│   CSS Grid + 绝对定位    │                 │   → tasks + workload     │
│   缩放: 日视图/周视图    │                 │                          │
│                         │                 │ detectDelays()           │
│ ScheduleStats           │ ──GET /delays──→ │ detectConflicts()        │
│   6 个统计卡片 + 工作量图│ ──GET /conflicts→│ insertionSimulation()    │
│                         │                 │                          │
│ InsertionDialog         │ ──POST /insertion→│ (纯计算，对比两次排期)    │
│   表单 → 结果展示        │                 └──────────────────────────┘
│                         │
│ AI 排期助手             │ ──AI Tools────→ │ schedule-advice.ts       │
│   5 个预设问题           │                 │ rebalance-suggest.ts     │
│   自由提问              │                 │ insertion-evaluation.ts   │
└─────────────────────────┘                 └──────────────────────────┘
```

### 4.3 甘特图渲染原理

GanttChart 是**纯手写组件**（未使用第三方甘特图库），实现方式：

```
容器: CSS Grid 布局
  左列 (200px): 任务名称 + 优先级标签
  右列 (剩余空间): 时间轴区域

时间轴:
  - 日期范围 = (最早日期 - 2天) 到 (最晚日期 + 2天)
  - 每天一列: 日视图 40px, 周视图 20px
  - 周末列加灰色背景

任务条:
  - 绝对定位, left = (任务开始日期 - 起始日期) × 每日宽度
  - width = (结束日期 - 开始日期) × 每日宽度
  - 颜色按优先级: 红(URGENT) / 橙(HIGH) / 蓝(MEDIUM) / 灰(LOW)
  - 延期任务强制红色, 冲突任务加橙色边框

辅助标记:
  - 今日线: 紫色竖线 + "今日"标签
  - 截止日: 红色小圆点竖线
  - 底部: 每日工时数, 超载日红色显示
```

### 4.4 AI 集成

排期数据通过三个 AI 工具暴露给 AI 助手：

| 工具 | 功能 | 典型提问 |
|------|------|---------|
| `get_schedule_advice` | 排期建议 + 优先级打分 | "我应该先做什么？" "能在周五前完成吗？" |
| `suggest_rebalance` | 冲突解决 + 延期处理建议 | "排期有什么问题？" "怎么调整？" |
| `evaluate_insertion` | 插单影响评估 | "加一个 8 小时的任务会怎样？" |

AI 侧边栏的「排期」标签页提供 5 个一键问题：
1. **延期任务** — 列出所有过期未完成的任务
2. **今日焦点** — 今天应该做什么
3. **排期健康度** — 整体排期是否有风险
4. **插单评估** — 评估新增任务的影响
5. **什么时候有空** — 查找最近的空闲时间段

### 4.5 每日提醒 (Cron Job)

```
文件: backend/src/jobs/due-reminder.job.ts
触发: 每天 08:00 (Asia/Shanghai)
逻辑:
  for 每个开启了任务提醒的用户:
    for 该用户的每个活跃项目:
      调用 detectDelays(projectId)
      if 有过期任务:
        创建 TASK_DUE 类型通知 → 推送到前端
```

## 六、设计决策

**为什么实时计算而不存储？**
- 排期完全依赖任务数据，任务一变排期就变
- 避免了数据不一致的问题
- 算法是纯函数，每次计算结果确定，可缓存

**为什么手写甘特图而不用库？**
- 需求明确（单项目、纯工时分配），不需要复杂库的全部功能
- 完全控制样式和交互，与设计系统一致
- 包体积更小，无额外依赖

**为什么排序用 "优先级 > 截止日期 > 工时" ？**
- 优先级高的任务必须先排（业务逻辑）
- 同优先级时，截止日期早的先排（避免延期）
- 同截止日期时，工时短的先排（快速完成，释放产能）
