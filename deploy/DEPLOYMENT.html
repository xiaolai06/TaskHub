🔀 Nginx 反向代理

配置文件位置：/etc/nginx/sites-enabled/taskflow

nginx# HTTP → HTTPS 自动跳转
server {
    listen 80;
    server_name xiaolai.duckdns.org 47.106.109.59;
    return 301 https://$host$request_uri;
}

# HTTPS 主配置
server {
    listen 443 ssl;
    server_name xiaolai.duckdns.org 47.106.109.59;
    client_max_body_size 10m;

    # SSL 证书
    ssl_certificate     /etc/letsencrypt/live/xiaolai.duckdns.org/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/xiaolai.duckdns.org/privkey.pem;

    # 后端 API
    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_buffering off;  # SSE 流式响应必须关闭
    }

    # 前端（Next.js）
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
⚠️ proxy_buffering off AI 对话和语音识别使用 SSE（Server-Sent Events）流式传输。如果 Nginx 开启 buffering，AI 回复会等全部生成完才一次性返回，用户看不到逐字输出效果。 
测试并重载sudo nginx -t && sudo systemctl reload nginx

🛡️ CORS 策略

backend/src/app.tsapp.use(cors({
  origin: (origin, callback) => {
    // 无 origin（同源/curl）→ 放行
    if (!origin) { callback(null, true); return; }

    // 开发环境：允许 localhost
    if (config.nodeEnv !== 'production') {
      if (origin.includes('localhost') || origin.includes('127.0.0.1')) {
        callback(null, true); return;
      }
    }

    // 生产环境：只允许 FRONTEND_URL
    const allowed = config.frontendUrl.split(',')
      .some(u => origin === u.trim());
    callback(null, allowed ? origin : false);
  },
  credentials: true,
}));
场景开发环境生产环境
允许的来源localhost:*, 127.0.0.1:*https://xiaolai.duckdns.org
无 Origin（curl）✅ 放行✅ 放行
其他来源✅ 放行❌ 拒绝

⚡ Next.js 配置差异

frontend/next.config.tsimport type { NextConfig } from 'next';

const isDev = process.env.NODE_ENV !== 'production';

const nextConfig: NextConfig = {
  turbopack: { root: process.cwd() },
  // 仅开发环境使用 rewrite 代理 API
  // 生产环境由 Nginx 处理，不需要 rewrite
  ...(isDev && {
    async rewrites() {
      return [{
        source: '/api/:path*',

      }];
    },
  }),
};

export default nextConfig;
✅ 关键设计 isDev 判断确保 rewrite 只在 next dev 时生效。next build 时 NODE_ENV=production，rewrite 不会写入构建产物，避免生产环境出现死循环代理。

🔥 实战踩坑录

以下是部署过程中实际遇到的 10 个问题，按严重程度分为三级。每个问题均记录了现象 → 原因 → 修复方案。

⚠️ 阅读提示 这些问题在本地开发环境中不会出现（或表现不同），只有真正部署到服务器、通过域名 HTTPS 访问时才会暴露。建议首次部署前通读本节。
1
API 地址硬编码导致服务器所有请求 404 前端 fetch 地址写死 localhost:3001，生产环境完全不可用
CRITICAL ▾

🔴 现象

部署到服务器后，登录页面可以打开，但输入账号密码点登录后无任何反应。浏览器控制台报 Failed to fetch 或 ERR_CONNECTION_REFUSED。所有 API 请求全部失败。

🔍 原因分析

  • 前端代码中多处 fetch 地址写成了 http://localhost:3001/api/xxx 绝对路径
  • 浏览器在用户电脑上运行，localhost 指向的是用户本机，而非服务器
  • 涉及文件：lib/api.ts、useAiChat.ts、useSpeechRecognition.ts、ChatInput.tsx、AiPanel.tsx 等
  • 本地开发时因 Next.js rewrite 代理恰好转发到同机 3001 端口，所以不会发现问题

✅ 修复方案

将所有前端 fetch 地址统一改为相对路径 /api，由环境差异自动适配：

修复前fetch('http://localhost:3001/api/projects')
修复后fetch('/api/projects')
// 开发: Next.js rewrite → localhost:3001
// 生产: Nginx location → 127.0.0.1:3001
💡 经验教训 前端代码中永远不要硬编码后端地址。统一使用相对路径 /api，配合环境的代理机制（开发用 Next.js rewrite，生产用 Nginx）自动转发。
参考：API 路径策略章节
2
Cookie secure 标志导致登录后立即 401 登录成功但后续请求全部返回未授权
CRITICAL ▾

🔴 现象

用户登录接口返回 200 成功，但浏览器没有保存 Cookie。后续所有 API 请求返回 401 Unauthorized，页面显示未登录状态。

🔍 原因分析

  • 后端 auth.routes.ts 中 Cookie 设置了 secure: true
  • secure: true 表示浏览器只在 HTTPS 连接下才保存和发送 Cookie
  • 如果服务器暂时通过 HTTP 访问（如还未配置 SSL），Cookie 完全不工作
  • 更隐蔽的是：sameSite: 'none' 配合 secure: false 会被浏览器拒绝

✅ 修复方案

将 secure 标志改为环境变量控制：

backend/.env# HTTP 环境设为 false，HTTPS 环境设为 true
COOKIE_SECURE=true
auth.routes.tsconst COOKIE_SECURE = process.env.COOKIE_SECURE === 'true';

res.cookie('token', token, {
  httpOnly: true,
  secure: COOKIE_SECURE,
  sameSite: COOKIE_SECURE ? 'none' : 'lax',
  maxAge: 7 * 24 * 60 * 60 * 1000,
  path: '/',
});
💡 经验教训 secure、sameSite 等 Cookie 属性必须与实际协议一致。HTTPS 环境必须 secure: true + sameSite: 'none'，HTTP 环境必须 secure: false + sameSite: 'lax'。用环境变量统一控制。
3
CORS 策略拦截生产环境所有请求 Access-Control-Allow-Origin 不匹配，浏览器直接拒绝响应
CRITICAL ▾

🔴 现象

浏览器控制台报 Access to fetch at 'https://xiaolai.duckdns.org/api/...' from origin 'https://xiaolai.duckdns.org' has been blocked by CORS policy。即使前后端在同一域名下也被拦截。

🔍 原因分析

  • 后端 CORS 配置中 FRONTEND_URL 环境变量未设置或值不正确
  • 配置写成了 http://localhost:3002（开发环境地址），与生产域名不匹配
  • 更深层问题：开发环境的 CORS 宽松策略（放行所有 localhost）掩盖了配置问题

✅ 修复方案

采用动态 origin 校验，区分开发/生产环境：

backend/.envFRONTEND_URL=https://xiaolai.duckdns.org
app.tsorigin: (origin, callback) => {
  if (!origin) { callback(null, true); return; }
  // 开发环境：允许 localhost
  if (config.nodeEnv !== 'production') {
    if (origin.includes('localhost') || origin.includes('127.0.0.1')) {
      callback(null, true); return;
    }
  }
  // 生产环境：只允许 FRONTEND_URL
  const allowed = config.frontendUrl.split(',')
    .some(u => origin === u.trim());
  callback(null, allowed ? origin : false);
}
💡 经验教训 CORS 是最容易被忽略的生产问题。开发时 localhost 宽松放行让你以为没问题，部署后才发现全线报错。FRONTEND_URL 必须与实际访问域名完全一致（含协议 https://）。

Next.js rewrites 在生产环境导致请求死循环 rewrite 与 Nginx 代理冲突，API 请求无限转发
HIGH ▾

🟡 现象

生产环境 API 请求超时，Nginx 错误日志出现 upstream timed out。PM2 后端日志中看不到任何请求到达。

🔍 原因分析

  • next.config.ts 中配置了 rewrite 将 /api/* 代理到 localhost:3001
  • next build 时如果 NODE_ENV 未正确设置为 production，rewrite 会写入构建产物
  • 请求流：浏览器 → Nginx(:443) → Next.js(:3000) → rewrite → Next.js(:3000) → ... 死循环
  • Nginx 的 location /api/ 应该直接转发到后端，但 rewrite 抢先拦截了

✅ 修复方案

用 isDev 判断条件性启用 rewrite：

next.config.tsconst isDev = process.env.NODE_ENV !== 'production';

const nextConfig: NextConfig = {
  ...(isDev && {
    async rewrites() {
      return [{
        source: '/api/:path*',
        destination: 'http://localhost:3001/api/:path*',
      }];
    },
  }),
};
💡 经验教训 Next.js 的 rewrites 在 build 时会写入 .next/routes-manifest.json。如果不在代码中加 isDev 守卫，生产环境的 next start 会执行 rewrite，与 Nginx 的 location /api/ 冲突。这种问题只在 Nginx 代理模式下才会暴露。

Nginx proxy_buffering 导致 AI 对话卡住无响应 SSE 流式响应被缓冲，用户看不到逐字输出
HIGH ▾

🟡 现象

点击 AI 对话发送按钮后，页面一直处于加载状态，等待 10-30 秒后突然一次性显示完整回复。本地开发时是逐字输出的流式效果，生产环境完全丢失。

🔍 原因分析

  • Nginx 默认开启 proxy_buffering on，会缓冲后端响应
  • SSE（Server-Sent Events）依赖持续的数据流推送，每推送一个 token 作为一个 event
  • Nginx 将这些小 event 缓存在内存中，直到后端响应结束才一次性转发给浏览器
  • 不仅影响 AI 对话，也影响语音识别的实时状态反馈

✅ 修复方案

在 Nginx 的 location /api/ 中关闭缓冲：

nginx.conflocation /api/ {
    proxy_pass http://127.0.0.1:3001;
    proxy_buffering off;  # ← 关键！SSE 流式必须关闭
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
💡 经验教训 任何使用 SSE 或 WebSocket 的后端接口，Nginx 代理都必须关闭 proxy_buffering。本地开发直连后端没有这个问题，只在 Nginx 代理层才会出现。这是 SSE 部署的「必踩坑」。

💡 经验教训 浏览器的 getUserMedia、SpeechRecognition、crypto.subtle 等 API 都要求安全上下文。开发时用 localhost 绕过了这个限制，部署到服务器后才发现。应提前检测 window.isSecureContext 并做降级处理。

MEDIUM ▾

🟠 现象

用户登录后正常使用，但几小时后突然被踢出（401）。再次登录又正常，几小时后再次被踢。表现为「间歇性登录失效」。

🔍 原因分析

  • middleware/auth.ts 会在 Token 剩余有效期不足 1 天时自动签发新 Token 并更新 Cookie
  • 续期代码中 Cookie 的 secure 标志写死了 true，没有与登录接口同步
  • 如果在 HTTP 环境调试（COOKIE_SECURE=false），登录接口设置的 Cookie 是 secure: false，但续期时覆盖为 secure: true
  • 浏览器发现同一个 Cookie 的 secure 属性变了，可能直接丢弃

middleware/auth.tsconst COOKIE_SECURE = process.env.COOKIE_SECURE === 'true';

// 续期时与登录接口保持一致
res.cookie('token', newToken, {
  httpOnly: true,
  secure: COOKIE_SECURE,
  sameSite: COOKIE_SECURE ? 'none' : 'lax',
  maxAge: 7 * 24 * 60 * 60 * 1000,
  path: '/',
});
💡 经验教训 设置 Cookie 的地方不止一处（登录接口、auth 中间件续期），所有设置点必须用同一份配置。建议提取为 cookieOptions() 工具函数，避免多处硬编码导致不一致。

语音识别多供应商降级逻辑不完善 STT 主服务不可用时，未正确切换到备选方案

  • speech.service.ts 中多供应商降级逻辑缺少 catch 分支
  • 主供应商超时或返回 500 时，异常被吞掉，没有触发下一个供应商的尝试

    if (result) return result;

    continue; // 尝试下一个供应商
  }
}
throw new Error('All STT providers failed');

🏗️ 多应用部署指南

一台服务器完全可以同时运行多个应用，核心原理是端口分配和 Nginx 路由分流。

💡 核心概念 服务器就像一栋写字楼，每个应用是不同的办公室。Nginx 是前台接待，根据「域名」或「路径」把访客引导到正确的办公室。只要每个应用占不同的端口，就能和平共处。

一、端口分配策略

每个应用需要前后端各一个端口，建议按应用编号递增：

应用前端端口后端端口说明
TaskFlow+30003001当前项目
应用 B30023003新部署的应用
应用 C30043005继续递增
应用 D30063007以此类推...
⚠️ 避开端口冲突 部署前先用 ss -tlnp | grep :端口号 检查端口是否已被占用。常见被占用端口：22（SSH）、80/443（Nginx）、5555（Prisma Studio）。

二、Nginx 多域名配置

有两种路由方式：按域名分流（推荐）或按路径分流。

方式 A：域名分流（推荐）

每个应用一个独立域名或子域名，配置最清晰：

nginx# 应用 1：TaskFlow+ （已有域名 xiaolai.duckdns.org）
server {
    listen 443 ssl;
    server_name xiaolai.duckdns.org;

    ssl_certificate     /etc/letsencrypt/live/xiaolai.duckdns.org/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/xiaolai.duckdns.org/privkey.pem;

    location /api/ {
        proxy_pass http://127.0.0.1:3001;       # TaskFlow 后端
        proxy_buffering off;
    }
    location / {
        proxy_pass http://127.0.0.1:3000;       # TaskFlow 前端
    }
}

# 应用 2：新应用（新域名 newapp.duckdns.org）
server {
    listen 443 ssl;
    server_name newapp.duckdns.org;

    ssl_certificate     /etc/letsencrypt/live/newapp.duckdns.org/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/newapp.duckdns.org/privkey.pem;

    location /api/ {
        proxy_pass http://127.0.0.1:3003;       # 新应用后端
        proxy_buffering off;
    }
    location / {
        proxy_pass http://127.0.0.1:3002;       # 新应用前端
    }
}
✅ 域名分流的优势
  • 配置独立，互不影响
  • SSL 证书独立申请
  • 可以单独重启某个应用
  • CORS 配置各自独立

方式 B：路径分流（同一域名）

如果只有一个域名，用路径前缀区分不同应用：

nginxserver {
    listen 443 ssl;
    server_name xiaolai.duckdns.org;

    ssl_certificate     /etc/letsencrypt/live/xiaolai.duckdns.org/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/xiaolai.duckdns.org/privkey.pem;

    # TaskFlow+（默认）
    location /api/ {
        proxy_pass http://127.0.0.1:3001;
    }
    location / {
        proxy_pass http://127.0.0.1:3000;
    }

    # 新应用（路径前缀 /newapp/）
    location /newapp/api/ {
        rewrite ^/newapp/(.*) /$1 break;   # 去掉 /newapp 前缀
        proxy_pass http://127.0.0.1:3003;
    }
    location /newapp/ {
        rewrite ^/newapp/(.*) /$1 break;
        proxy_pass http://127.0.0.1:3002;
    }
}
⚠️ 路径分流的坑
  • 需要前端代码支持 basePath（Next.js 的 basePath 配置）
  • Cookie 路径需要匹配，否则会串应用
  • 静态资源路径可能冲突
  • 能用域名分流就不用路径分流

📊 并发访问能力说明

你的 TaskFlow+ 网站能同时支持多少人访问？这取决于服务器配置、应用架构和访问模式三个维度。

💡 先说结论 以阿里云 ECS 1 核 2GB 配置为基准，TaskFlow+ 大约能支撑 50-100 人同时在线（常规 CRUD 操作），AI 对话场景下约 20-30 人同时对话。如果需要更大并发，升级服务器或开启 PM2 集群模式即可。

一、各组件并发能力分析

组件并发瓶颈当前配置下预估影响因素
Nginx 几乎不是瓶颈 数千连接 C10K 问题早已解决，Nginx 处理静态请求极快
Next.js 前端 内存 + SSR 渲染 100-200 并发 SSR 页面每次请求都要服务端渲染，消耗 CPU 和内存
Express 后端 单线程事件循环 200-500 并发 Node.js 单线程，但 I/O 操作异步非阻塞
SQLite 数据库 写锁（串行写入） ~50 并发写 SQLite 同一时刻只允许一个写操作，读操作不受影响
AI 对话（SSE） 长连接占用 + API 限频 20-30 并发 每个 AI 对话占用一个 TCP 连接直到回复完成，且受第三方 API 限频
⚠️ SQLite 写锁是最大瓶颈 SQLite 是文件级数据库，写操作会锁住整个数据库文件。多个用户同时创建/修改任务时会排队等待。对于中小团队（20 人以内）完全够用，超过 50 人建议考虑迁移到 PostgreSQL。

三、当前架构的瓶颈点

瓶颈原因影响解决方案
SQLite 写锁 单文件数据库，写操作排他锁 多人同时写入时短暂等待 团队 >50 人时迁移到 PostgreSQL
Node.js 单线程 后端只有 1 个进程处理请求 CPU 密集操作会阻塞其他请求 PM2 集群模式（-i max）

服务器内存 1-2GB 内存有限 高并发时 OOM 崩溃 升级服务器或优化内存使用

四、优化并发的方案（由易到难）

方案 1：PM2 集群模式（零代码改动）

让 Node.js 利用多个 CPU 核心，每个核心运行一个后端实例：

bash# 当前：单实例
pm2 start backend/dist/server.js --name taskflow-backend

# 优化：集群模式（自动使用所有 CPU 核心）
pm2 start backend/dist/server.js --name taskflow-backend -i max

# 如果是 2 核 CPU，就启动 2 个实例
# 4 核就启动 4 个实例
# PM2 自动做负载均衡
✅ 效果 后端并发能力翻倍（2 核 → 2x，4 核 → 4x）。零代码改动，只需改一行启动命令。这是性价比最高的优化。

方案 2：Nginx 缓存静态资源

减少前端 SSR 压力，静态页面直接由 Nginx 返回：

nginx# Next.js 静态资源缓存（.next/static/）
location /_next/static/ {
    proxy_pass http://127.0.0.1:3000;
    expires 365d;              # 缓存 1 年
    add_header Cache-Control "public, immutable";
}

# API 响应不缓存
location /api/ {
    proxy_pass http://127.0.0.1:3001;
    proxy_buffering off;
    add_header Cache-Control "no-store";  # 不缓存 API
}

方案 4：SQLite → PostgreSQL（大团队适用）

当团队超过 50 人、并发写入频繁时，考虑迁移数据库：

  • PostgreSQL 支持真正的并发写入，没有文件锁
  • Prisma 切换数据库只需改 DATABASE_URL 和 datasource provider
  • 需要额外安装 PostgreSQL 服务（或用云数据库如 Supabase 免费版）
